Chronicle 是什么？

OpenAI 于 2026-04-20 为 Codex 桌面端推出 Chronicle：后台 Agent 定期截取屏幕内容，通过视觉理解生成本地 Markdown 记忆文件，让编程助手在用户说"这个""刚才那个"时能自动关联上下文。

记忆文件存放在 ~/.codex/memories_extensions/chronicle/，用户可读、可编辑、可删除。

为什么这件事值得关注？

核心命题：编程助手正从"仅依赖聊天记录"进化到"理解整个桌面工作环境"。上下文获取的主动权，正从用户手中转移到 Agent 自身。

这指向一个快速成型的赛道 —— 屏幕感知型 AI 编程/工作助手（Screen-Aware AI Agent）。

三个竞品分别是谁？

① OpenAI Codex Chronicle — 后台截屏 → 视觉摘要 → 本地记忆，推理时自动引用。

② Anthropic Claude Code + Routines — 可复用自动化工作流 + Computer Use 直接操控桌面。

③ Screenpipe（MIT 开源） — 24/7 录屏 + 音频转写 → 全本地存储 → 作为 MCP Server 供各种 AI 助手查询。

💡 Core Learnings

点击卡片展开完整分析

1. 从"用户喂上下文"到"Agent 自己看"

上下文获取的主动权正从用户转移到 Agent。
👆 点击展开
Chronicle 代表的核心范式转移：过去用户需要手动粘贴代码、截图、日志；现在 Agent 通过屏幕感知自动建立理解。

启发：任何需要用户反复"交代背景"的 AI 功能，都值得思考能否让 Agent 被动感知。

2. 记忆架构正在分层

短期 → 中期 → 长期，三层记忆各司其职。
👆 点击展开
• 短期记忆：当前会话上下文
• 中期记忆：屏幕/工作流摘要（Chronicle、Screenpipe）
• 长期记忆：用户偏好、项目约定（Codex Memories、CLAUDE.md）

启发：AI 工具设计应考虑分层架构，而非把所有上下文塞进同一个 prompt。

3. 隐私 vs 实用性 — 产品化核心挑战

云端处理强但合规受限，全本地隐私好但门槛高。
👆 点击展开
Chronicle 选择"云端处理 + 本地存储"的折中，但 EU 直接不可用；Screenpipe 全本地但牺牲易用性。

启发：如果要做"屏幕感知"能力，需要在数据驻留、加密、合规审批上提前投入。

4. Rate Limit 经济学

"持续感知"是有成本的，需要精算 ROI。
👆 点击展开
Chronicle 的后台 Agent 会持续消耗 API 额度，即使用户没有主动提问。

启发：企业级部署时，需要精确核算被动感知的 ROI vs. 用户主动提供上下文的成本差。

🔥 对运营中台的启发

"Agent 自己看屏幕"这个范式，翻译到我们的场景就是——能不能让 Tiko 自己看商家在干什么，让 Pearl 自动记住运营刚才在哪个系统看了什么？

我们负责 TikTok Shop 运营中台，核心产品：Pearl 运营工作台 + Tiko AI 助手。下面从三个角度展开——当前痛点怎么解、ROI 不高怎么办、以及更大的想象空间。

🤖 Tiko 的核心痛点：商家说"刚才"，Tiko 不知道"刚才"

商家问"我刚才改了物流模板怎么没生效"——Tiko 只能追问。如果它能像 Chronicle 一样"看到"商家的操作呢？
👆 点击展开
Tiko 今天处理商家问题的流程是：商家描述问题 → Tiko 检索知识库 → 调用垂域 Skill 查数据 → 给出回答。这个流程的瓶颈在第一步：商家往往描述不清楚。

典型 case："我刚才设置了一下运费模板，怎么前台没变？"——Tiko 不知道商家改了哪个模板、改了什么字段、改之前是什么状态。只能反复追问，体验差，ORR 掉。

Chronicle 给的启发：问题不在 Tiko 的知识不够，而在它缺少「操作现场」。如果在商家发起会话时，自动注入最近 10 分钟的关键操作序列——不是录屏，只是结构化的操作日志（"17:32 进入物流模板页 → 17:33 修改模板 A 配送区域 → 17:34 点击保存"），Tiko 就能直接定位问题。

💡 怎么做

  • 新增一个"操作上下文"Skill：会话开启时自动触发，从 Seller Center 操作事件流拉取最近操作。只做事件驱动，不做 always-on，成本增量约 500 tokens/会话。
  • 操作-问题关联模型：不是所有操作都有用。用一个轻量分类器判断"商家当前问题"和"哪些最近操作"有关联，只召回相关的。
  • 效果可验证：上线后用现有的 ORR 评测 + Bad Case Review 机制跟踪"追问轮次减少率"和"首次解决率变化"。

🖥️ Pearl 的隐形杀手：运营每天浪费多少时间"回忆刚才在干嘛"

运营在 Pearl、数据看板、飞书、工单系统之间频繁切换。每次切回 Pearl，脑子是空的。
👆 点击展开
一个 AM 的典型半小时：在 Pearl 查商家 A 的 GMV 数据 → 切到 T2T 报告看竞品对比 → 切到飞书跟运营 lead 讨论策略 → 切回 Pearl 准备执行。问题来了：执行什么？刚才看的是哪个商家？那个数据指标是多少来着？

这不是 Pearl 功能缺失，是跨系统上下文断裂。Chronicle 的"记忆分层"给了直接方案：

💡 "操作回忆"能力

  • 短期记忆：Pearl 当前页面状态（已有）。
  • 中期记忆（缺失）：运营最近 30 分钟的操作轨迹摘要。切回 Pearl 时，AI 助手一句话告诉你"你刚才在看商家 A 的 GMV 下滑，对比了 3 个竞品店铺，在飞书跟 XX 讨论了促销策略"。
  • 长期记忆：运营的常用策略偏好和历史决策模式（可基于 Pearl 操作日志积累）。

这个能力不需要录屏，Pearl 本身的页面访问日志 + 数据看板的查询记录 + 飞书消息摘要就够了。

💰 ROI 不高怎么办？三个"聪明地省钱"的策略

如果像 Chronicle 一样全量感知操作，数十万商家的推理成本根本扛不住。但不感知又回到老路。
👆 点击展开
核心矛盾：全量感知太贵，不感知体验差。解法不是"做不做"的二选一，而是"什么时候做、对谁做、做多深"。

🔧 策略一：事件驱动，不是轮询

  • 不 24/7 监控所有商家。只在关键事件发生时回溯操作：商家发起咨询、设置变更失败、订单异常。
  • 类比：从"每秒截屏"变成"关键动作才拍照"。推理成本降 90%+。

🔧 策略二：分层压缩，热温冷三级

  • 热层（5 分钟内）：完整操作序列直接注入。温层（5-60 分钟）：小模型压缩成 1-2 句话。冷层（1h+）：只留关键词索引，按需检索。
  • 每次对话上下文增量控制在 500-2000 tokens，成本约等于多一个知识库召回。

🔧 策略三：让数据告诉你谁值得 always-on

  • 初期对所有场景用最小感知。当 Tiko 答不好时，用户点"不满意"触发深度回溯。
  • 系统统计哪些品类 × 问题类型的回溯被频繁触发 → 逐步对这些场景自动开启感知。
  • ROI 通过实际使用动态优化，不拍脑袋。

🧠 更大的想象空间

跳出"操作记忆"框架，这个趋势还能怎么用？
👆 点击展开

💡 从被动客服到预判式助手

  • Chronicle 是"看到你在做什么，下次你问我已经懂了"。Tiko 可以更进一步——"看到你卡住了，我主动来帮你"。
  • 具体：监控 Seller Center 操作模式，检测到异常（反复修改同一设置、某页面停留异常久），Tiko 主动弹出引导。
  • 这本质上是把 Tiko 从"客服"升级成"运营教练"——不等你来问，看到你犯错就提醒。对新手商家和新入职 AM 的价值尤其大。

💡 运营 Routine：从"人学工具"到"工具学人"

  • Claude Code Routines 的思路：把高频操作封装成可复用工作流。运营人员每周一做 GMV 巡检、每天看商家告警、周五写周报——这些都可以变成运营 Routine。
  • Pearl AI Copilot 观察操作模式，自动生成 Routine 模板："检测到你每周一都做这套巡检，要不要我帮你自动化？"
  • 终极形态：Pearl 不是运营"用的工具"，是"跟运营一起干活的搭档"——它知道你的习惯、记得你的策略、能替你执行重复动作。

💡 从单商家对话到跨商家知识飞轮

  • Screenpipe 的"全量记录+检索"映射过来：A 商家遇到的物流配置坑，解决方案自动沉淀到知识库，B 商家遇到同样问题时 Tiko 直接引用。
  • 目前知识库人工维护 + 自动提取并行，但真实对话中有大量"只在实际操作中才会遇到的 edge case"没被覆盖。如果能从对话中自动挖掘和沉淀，知识库的活性和覆盖率会质变。

📌 行动建议

短期（1-2 月）：Tiko 新增"操作上下文 Skill"，会话触发 + 按需回溯，预计减少追问轮次 30%+。

中期（1 季度）：Pearl 上线"操作回忆"摘要能力。Tiko 基于 Bad Case 数据验证哪些场景值得 always-on 感知。

长期（2-3 季度）：运营 Routine 自动化框架 + 跨商家知识飞轮，从工具进化为 AI 搭档。
