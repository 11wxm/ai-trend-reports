Chronicle 是什么？

OpenAI 于 2026-04-20 为 Codex 桌面端推出 Chronicle：后台 Agent 定期截取屏幕内容，通过视觉理解生成本地 Markdown 记忆文件，让编程助手在用户说"这个""刚才那个"时能自动关联上下文。

记忆文件存放在 ~/.codex/memories_extensions/chronicle/，用户可读、可编辑、可删除。

为什么这件事值得关注？

核心命题：编程助手正从"仅依赖聊天记录"进化到"理解整个桌面工作环境"。上下文获取的主动权，正从用户手中转移到 Agent 自身。

这指向一个快速成型的赛道 —— 屏幕感知型 AI 编程/工作助手（Screen-Aware AI Agent）。本报告将 Chronicle 与两个核心竞品进行深度横向对比。

三个竞品分别是谁？

① OpenAI Codex Chronicle — 后台截屏 → 视觉摘要 → 本地记忆，推理时自动引用。

② Anthropic Claude Code + Routines — 可复用自动化工作流 + Computer Use 直接操控桌面。

③ Screenpipe（MIT 开源） — 24/7 录屏 + 音频转写 → 全本地存储 → 作为 MCP Server 供各种 AI 助手查询。

Chronicle 设置中开关即可，但会消耗 rate limit。Claude Code Routines 需编写 YAML 配置并理解触发机制。Screenpipe 需本地安装、配置 Pipe 插件，有一定工程门槛。

Core Learnings

点击卡片展开完整分析

1. 交互范式：从"用户喂上下文"到"Agent 自己看"

上下文获取的主动权正从用户转移到 Agent。
👆 点击展开
Chronicle 代表的核心范式转移：过去用户需要手动粘贴代码、截图、日志；现在 Agent 通过屏幕感知自动建立工作环境理解。

启发：任何需要用户反复"交代背景"的 AI 功能，都值得思考能否让 Agent 被动感知，减少用户的上下文搬运成本。

2. 记忆架构正在分层

短期 → 中期 → 长期，三层记忆各司其职。
👆 点击展开
赛道内出现清晰的记忆分层模式：
• 短期记忆：当前会话上下文（所有产品都有）
• 中期记忆：屏幕 / 工作流摘要（Chronicle、Screenpipe）
• 长期记忆：用户偏好、项目约定（Codex Memories、CLAUDE.md）

启发：AI 工具设计应考虑这种分层架构，而非把所有上下文塞进同一个 prompt。

3. 隐私 vs 实用性 — 产品化核心挑战

云端处理强但合规受限，全本地隐私好但门槛高。
👆 点击展开
Chronicle 选择"云端处理 + 本地存储"的折中，但 EU 直接不可用；Screenpipe 全本地但牺牲易用性。

启发：如果要做类似"屏幕感知"能力，需要在数据驻留、加密、合规审批上提前投入。企业场景中这个张力更突出。

4. Rate Limit 经济学

"持续感知"是有成本的，需要精算 ROI。
👆 点击展开
Chronicle 的后台 Agent 会持续消耗 API 额度，即使用户没有主动提问。

启发：企业级部署时，需要精确核算被动感知的 ROI vs. 用户主动提供上下文的成本差。不是所有场景都适合 always-on 的屏幕感知。

跟运营中台的关系

落到 TikTok Shop 运营中台 · Pearl 平台 · Tiko 机器人，这些趋势对我们意味着什么？点击卡片展开分析。

业务背景：我们负责 TikTok Shop 运营中台，核心产品是 Pearl（运营工作台）和 Tiko（智能客服/运营机器人）。运营人员日常在 Pearl 后台、数据看板、飞书、工单系统之间频繁切换，Tiko 则需要理解商家历史工单和运营策略来自动化回复。

🤖 Tiko：从"商家喂工单"到"Tiko 自己看操作记录"

Chronicle 的"屏幕感知"范式，映射到 Tiko 就是"操作记录感知"。
👆 点击展开
目前 Tiko 回答商家问题时，主要依赖当前会话上下文和知识库检索。但商家的真实问题往往涉及之前的操作——比如"我刚才改了一下物流模板怎么没生效"。

如果 Tiko 能感知商家在 Seller Center 的最近操作序列（类似 Chronicle 的屏幕记忆），就可以直接关联到"你 5 分钟前修改了物流模板 A 的 XX 字段"，省去商家来回描述的成本。

可落地方向：Tiko 接入 Seller Center 操作日志作为"中期记忆"层，在会话中自动引用商家最近的关键操作。

🖥️ Pearl：运营人员的"跨系统上下文自动关联"

运营在 Pearl、数据看板、飞书、工单之间频繁切换，上下文丢失是最大痛点。
👆 点击展开
运营人员典型工作流：在 Pearl 后台查看商家数据 → 切到数据看板对比指标 → 切到飞书和 AM 讨论策略 → 回到 Pearl 执行操作。每次切换都丢失上下文，需要大量心智重建。

Chronicle 的"记忆分层"思路可以直接借鉴：
• 短期：Pearl 当前页面状态
• 中期：最近 30 分钟跨系统操作摘要（类似 Chronicle 的屏幕记忆）
• 长期：运营人员的常用策略偏好和历史决策模式

可落地方向：Pearl 的 AI 助手（如果有）可以自动汇总运营人员跨系统的操作轨迹，在切换回 Pearl 时提供上下文恢复摘要。

💰 成本与合规：always-on 感知在我们的场景可行吗？

持续感知的 ROI 核算 + 电商数据合规特殊性。
👆 点击展开
Chronicle 的 always-on 模式消耗大量 API 额度。对于 Tiko 这样的高并发场景（数十万商家同时在线），全量感知操作记录的推理成本会非常高。

更现实的路径是事件驱动 + 按需感知：只在商家发起咨询时回溯最近操作，而非 24/7 持续录制。这既控制成本，也规避了持续采集操作数据带来的合规风险（电商场景涉及交易数据、商家隐私）。

结论：Chronicle 的 always-on 模式不适合直接照搬，但"操作记忆 + 按需检索"的轻量版完全可以成为 Tiko 和 Pearl 的下一个能力升级点。

📌 对运营中台的一句话总结

"Agent 自己看屏幕"映射到我们的业务就是"Tiko 自己看操作记录 + Pearl 自动恢复跨系统上下文"。不需要 always-on，用事件驱动的轻量版"操作记忆"就能显著减少商家和运营的上下文搬运成本。
