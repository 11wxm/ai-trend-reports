🔍 研究缘起

点击卡片展开详细背景

Chronicle 是什么？

OpenAI 于 2026-04-20 为 Codex 桌面端推出 Chronicle：后台 Agent 定期截取屏幕内容，通过视觉理解生成本地 Markdown 记忆文件，让编程助手在用户说"这个""刚才那个"时能自动关联上下文。

记忆文件存放在 ~/.codex/memories_extensions/chronicle/，用户可读、可编辑、可删除。

为什么这件事值得关注？

核心命题：编程助手正从"仅依赖聊天记录"进化到"理解整个桌面工作环境"。上下文获取的主动权，正从用户手中转移到 Agent 自身。

这指向一个快速成型的赛道 —— 屏幕感知型 AI 编程/工作助手（Screen-Aware AI Agent）。本报告将 Chronicle 与两个核心竞品进行深度横向对比。

三个竞品分别是谁？

① OpenAI Codex Chronicle — 后台截屏 → 视觉摘要 → 本地记忆，推理时自动引用。

② Anthropic Claude Code + Routines — 可复用自动化工作流 + Computer Use 直接操控桌面。

③ Screenpipe（MIT 开源） — 24/7 录屏 + 音频转写 → 全本地存储 → 作为 MCP Server 供各种 AI 助手查询。

💡 Core Learnings

点击卡片展开完整分析

1. 交互范式：从"用户喂上下文"到"Agent 自己看"

上下文获取的主动权正从用户转移到 Agent。
👆 点击展开
Chronicle 代表的核心范式转移：过去用户需要手动粘贴代码、截图、日志；现在 Agent 通过屏幕感知自动建立工作环境理解。

启发：任何需要用户反复"交代背景"的 AI 功能，都值得思考能否让 Agent 被动感知，减少用户的上下文搬运成本。

2. 记忆架构正在分层

短期 → 中期 → 长期，三层记忆各司其职。
👆 点击展开
赛道内出现清晰的记忆分层模式：
• 短期记忆：当前会话上下文（所有产品都有）
• 中期记忆：屏幕 / 工作流摘要（Chronicle、Screenpipe）
• 长期记忆：用户偏好、项目约定（Codex Memories、CLAUDE.md）

启发：AI 工具设计应考虑这种分层架构，而非把所有上下文塞进同一个 prompt。

3. 隐私 vs 实用性 — 产品化核心挑战

云端处理强但合规受限，全本地隐私好但门槛高。
👆 点击展开
Chronicle 选择"云端处理 + 本地存储"的折中，但 EU 直接不可用；Screenpipe 全本地但牺牲易用性。

启发：如果要做类似"屏幕感知"能力，需要在数据驻留、加密、合规审批上提前投入。企业场景中这个张力更突出。

4. Rate Limit 经济学

"持续感知"是有成本的，需要精算 ROI。
👆 点击展开
Chronicle 的后台 Agent 会持续消耗 API 额度，即使用户没有主动提问。

启发：企业级部署时，需要精确核算被动感知的 ROI vs. 用户主动提供上下文的成本差。不是所有场景都适合 always-on 的屏幕感知。

🏪 跟运营中台的关系

落到 TikTok Shop 运营中台 · Pearl 平台 · Tiko 机器人，这些趋势对我们意味着什么？

业务背景：我们负责 TikTok Shop 运营中台，核心产品是 Pearl（运营工作台）和 Tiko（智能客服/运营机器人）。运营人员日常在 Pearl 后台、数据看板、飞书、工单系统之间频繁切换，Tiko 则需要理解商家历史工单和运营策略来自动化回复。

🤖 Tiko：从"商家喂工单"到"Tiko 自己看操作记录"

Chronicle 的"屏幕感知"范式，映射到 Tiko 就是"操作记录感知"。
👆 点击展开
目前 Tiko 回答商家问题时，主要依赖当前会话上下文和知识库检索。但商家的真实问题往往涉及之前的操作——比如"我刚才改了一下物流模板怎么没生效"。

如果 Tiko 能感知商家在 Seller Center 的最近操作序列（类似 Chronicle 的屏幕记忆），就可以直接关联到"你 5 分钟前修改了物流模板 A 的 XX 字段"，省去商家来回描述的成本。

💡 具体落地方案

  • 接入 Seller Center 操作日志：Tiko 会话开启时，自动拉取该商家最近 30 分钟的关键操作事件（页面跳转、设置变更、订单操作等），作为"中期记忆"注入对话上下文。
  • 操作-问题关联模型：训练一个轻量分类器，识别"商家当前问题"与"最近操作"的关联度。例如商家问物流问题，自动召回最近的物流设置变更操作。
  • 成本可控：不是 always-on 录制，而是会话触发 + 按需回溯。仅在商家发起对话时查询最近操作，单次增加约 500-1000 tokens 的上下文，推理成本增量极低。

🖥️ Pearl：运营人员的"跨系统上下文自动关联"

运营在 Pearl、数据看板、飞书、工单之间频繁切换，上下文丢失是最大痛点。
👆 点击展开
运营人员典型工作流：在 Pearl 后台查看商家数据 → 切到数据看板对比指标 → 切到飞书和 AM 讨论策略 → 回到 Pearl 执行操作。每次切换都丢失上下文，需要大量心智重建。

Chronicle 的"记忆分层"思路可以直接借鉴：
• 短期：Pearl 当前页面状态
• 中期：最近 30 分钟跨系统操作摘要
• 长期：运营人员的常用策略偏好和历史决策模式

💡 具体落地方案

  • "操作回忆"侧边栏：Pearl 内嵌一个 AI 侧边栏，自动汇总运营人员过去 1 小时内在 Pearl + 数据看板的操作轨迹，切换回来时一句话告诉你"你刚才在看 XX 商家的 GMV 下滑数据，对比了 3 个竞品"。
  • 策略推荐引擎：基于运营人员的历史决策模式（长期记忆），在相似场景出现时主动推荐策略。比如"上次遇到类似 GMV 下滑，你执行了 XX 促销策略，效果提升 15%"。
  • 跨系统剪贴板：运营在数据看板复制的数据、在飞书讨论的结论，自动同步到 Pearl 的 AI 助手，切换回来不丢失上下文。

💰 ROI 不高怎么办？—— 持续感知的成本优化方案

如果 always-on 感知的推理成本 ROI 不高，有哪些更聪明的做法？
👆 点击展开
Chronicle 的 always-on 模式消耗大量 API 额度。对于 Tiko 这样的高并发场景（数十万商家同时在线），全量感知操作记录的推理成本会非常高。
核心问题：如果"持续感知"的 ROI 不高——即大部分时候 Agent 感知到的上下文并没有被用到——怎么办？

🔧 方案 1：事件驱动替代轮询

  • 不做 24/7 持续截屏/录制，而是监听关键事件：设置变更、订单状态异常、商家发起咨询。
  • 只在事件触发时回溯最近操作记录，推理成本降低 90%+。
  • 类比：从"每秒截屏"变成"关键动作才拍照"。

🔧 方案 2：分层缓存 + 摘要压缩

  • 热层（5 分钟内）：完整操作序列，直接注入上下文。
  • 温层（5-60 分钟）：用小模型摘要为 1-2 句话，节省 tokens。
  • 冷层（1 小时以上）：仅保留操作关键词索引，按需检索。
  • 这套架构让每次对话的上下文增量控制在 500-2000 tokens，成本几乎可忽略。

🔧 方案 3：用户反馈驱动的自适应感知

  • 初期默认最小感知（仅当前页面状态），当 Tiko 答不好时，用户点"不满意"触发深度回溯。
  • 系统记录哪些品类/问题类型的"回溯"被频繁触发，逐步对这些场景自动开启感知。
  • 好处：让数据告诉你哪些场景值得"always-on"，避免拍脑袋。ROI 通过实际使用动态优化。

🧠 更大的想象空间：还能从这个趋势借鉴什么？

跳出"操作记忆"，这个"Agent 自主感知"的范式还能怎么用？
👆 点击展开

💡 想法 1：Tiko "预判式"客服

  • 不等商家来问，Tiko 主动发现问题。监控商家 Seller Center 的操作模式，检测到异常操作（比如反复修改同一个设置、在某个页面停留过长），主动弹出"需要帮助吗？"。
  • 类比 Chronicle：Chronicle 是"看到你在做什么，下次你问的时候我已经懂了"；Tiko 可以是"看到你在卡住，我主动来帮你"。
  • 价值：从被动客服变成主动运营助手，NPS 和解决率双提升。

💡 想法 2：Pearl "运营 Copilot" —— 从工具到搭档

  • 借鉴 Claude Code Routines 的"可复用工作流"概念：运营人员的日常操作（巡检商家数据 → 发现异常 → 执行策略 → 记录结果）可以封装为运营 Routine。
  • Pearl AI Copilot 观察运营人员的操作模式，自动生成 Routine 模板："检测到你每周一都做 XX 巡检，要不要我帮你自动化这个流程？"
  • 终极目标：把 Pearl 从"运营用的工具"升级为"跟运营一起干活的 AI 搭档"。

💡 想法 3：跨商家知识迁移

  • Screenpipe 的"全量记录 + 检索"思路可以映射到跨商家知识库：A 商家遇到的物流设置问题，解决方案自动沉淀，B 商家遇到类似问题时 Tiko 直接引用。
  • 目前 Tiko 的知识库是人工维护的，如果能从真实对话中自动提取和沉淀解决方案（类似 Chronicle 自动生成记忆），知识库的更新频率和质量都会大幅提升。
  • 关键指标：知识库覆盖率提升，首次解决率提升。

💡 想法 4：合规友好的"操作影子"

  • 电商场景涉及交易数据和商家隐私，不能像 Screenpipe 那样全量录制。但可以做"操作影子"：只记录操作类型和对象 ID，不记录具体数值。
  • 例如记录"商家 A 修改了物流模板 B 的配送区域字段"，但不记录具体改成了什么值。需要时通过 ID 实时查询最新状态。
  • 好处：既获得操作上下文的关联能力，又避免存储敏感交易数据，合规风险最小化。

📌 对运营中台的行动建议

短期（1-2 个月）：Tiko 接入 Seller Center 操作日志，实现"会话触发 + 按需回溯"的轻量操作记忆，预计可提升首次解决率 10-15%。

中期（1 个季度）：Pearl 上线"操作回忆"侧边栏 + 策略推荐引擎，减少运营的上下文重建时间。同时用反馈数据验证哪些场景值得 always-on 感知。

长期（2-3 个季度）：构建"运营 Routine"自动化框架 + 跨商家知识自动沉淀，把 Tiko 和 Pearl 从工具升级为 AI 搭档。
