Chronicle 是什么？

OpenAI 于 2026-04-20 为 Codex 桌面端推出 Chronicle：后台 Agent 定期截取屏幕内容，通过视觉理解生成本地 Markdown 记忆文件，让编程助手在用户说"这个""刚才那个"时能自动关联上下文。

记忆文件存放在 ~/.codex/memories_extensions/chronicle/，用户可读、可编辑、可删除。

为什么这件事值得关注？

核心命题：编程助手正从"仅依赖聊天记录"进化到"理解整个桌面工作环境"。上下文获取的主动权，正从用户手中转移到 Agent 自身。

这指向一个快速成型的赛道 —— 屏幕感知型 AI 编程/工作助手（Screen-Aware AI Agent）。

三个竞品分别是谁？

① OpenAI Codex Chronicle — 后台截屏 → 视觉摘要 → 本地记忆，推理时自动引用。

② Anthropic Claude Code + Routines — 可复用自动化工作流 + Computer Use 直接操控桌面。

③ Screenpipe（MIT 开源） — 24/7 录屏 + 音频转写 → 全本地存储 → 作为 MCP Server 供各种 AI 助手查询。

💡 Core Learnings

点击卡片展开完整分析

1. 从"用户喂上下文"到"Agent 自己看"

上下文获取的主动权正从用户转移到 Agent。
👆 点击展开
Chronicle 代表的核心范式转移：过去用户需要手动粘贴代码、截图、日志；现在 Agent 通过屏幕感知自动建立理解。

启发：任何需要用户反复"交代背景"的 AI 功能，都值得思考能否让 Agent 被动感知。

2. 记忆架构正在分层

短期 → 中期 → 长期，三层记忆各司其职。
👆 点击展开
• 短期记忆：当前会话上下文
• 中期记忆：屏幕/工作流摘要（Chronicle、Screenpipe）
• 长期记忆：用户偏好、项目约定（Codex Memories、CLAUDE.md）

启发：AI 工具设计应考虑分层架构，而非把所有上下文塞进同一个 prompt。

3. 隐私 vs 实用性 — 产品化核心挑战

云端处理强但合规受限，全本地隐私好但门槛高。
👆 点击展开
Chronicle 选择"云端处理 + 本地存储"的折中，但 EU 直接不可用；Screenpipe 全本地但牺牲易用性。

启发：如果要做类似"屏幕感知"能力，需要在数据驻留、加密、合规审批上提前投入。

4. Rate Limit 经济学

"持续感知"是有成本的，需要精算 ROI。
👆 点击展开
Chronicle 的后台 Agent 会持续消耗 API 额度，即使用户没有主动提问。

启发：企业级部署时，需要精确核算被动感知的 ROI vs. 用户主动提供上下文的成本差。

🔥 对运营中台的启发

从"屏幕感知 Agent"趋势出发，落到 TikTok Shop 运营中台的 Pearl 和 Tiko，这些能力可以怎么借鉴？

业务背景：我们负责 TikTok Shop 运营中台，核心产品是 Pearl（运营工作台）和 Tiko（TikTok Shop AI 助手）。

Tiko 是什么：一套面向 TikTok Shop AM 和商家的 AI 助手体系，以飞书 Bot + 社群助手 + Web 端为入口，以 MCP/Skills 生态 + 垂域知识库 + Agent 框架为引擎。当前已具备经营分析（店铺 GMV 诊断）、GNE 治理合规（罚单查询/CCR 分析）、全场景知识问答（10+ 领域覆盖）、50+ MCP Skills 等核心能力，重点攻坚 GNE 治理、经营分析、响应速度三大方向。

📚 TTS运营平台产品知识库 →

🤖 Tiko：从"商家喂工单"到"Tiko 主动感知操作上下文"

Chronicle 的"屏幕感知"映射到 Tiko = "操作记录感知"。Tiko 已有 50+ MCP Skills，但缺少操作上下文层。
👆 点击展开
当前痛点：Tiko 回答商家问题时，主要依赖当前会话上下文 + 知识库检索 + MCP Skills 调用。但商家的真实问题往往涉及刚才的操作——比如"我刚才改了物流模板怎么没生效"、"为什么我的 CCR 突然升了"。Tiko 缺少对商家最近操作序列的感知，只能反复追问。

💡 落地方案：Seller Center 操作日志接入

  • 接入操作事件流：Tiko 会话开启时，自动拉取该商家最近 30 分钟的关键操作（页面跳转、设置变更、订单操作），作为"中期记忆"注入上下文。
  • 操作-问题关联 Skill：新增一个 MCP Skill，根据商家当前问题自动召回相关操作。商家问物流问题 → 召回最近的物流设置变更。可复用 Tiko 现有的 Skills 召回 + 评测准出机制。
  • 成本可控：会话触发 + 按需回溯，不是 always-on。单次增加 500-1000 tokens，推理成本增量极低。
  • 预期效果：减少 30-50% 的"追问轮次"，提升首次解决率（ORR），直接改善 AM 和商家体验。

🖥️ Pearl：运营人员的"跨系统上下文自动关联"

运营在 Pearl、数据看板、飞书、工单之间频繁切换，上下文丢失是最大痛点。
👆 点击展开
运营人员典型工作流：Pearl 后台查商家数据 → 切到数据看板对比指标 → 切到飞书和 AM 讨论策略 → 回到 Pearl 执行操作。每次切换都丢失上下文。

Chronicle 的"记忆分层"思路可直接借鉴：
• 短期：Pearl 当前页面状态
• 中期：最近 30 分钟跨系统操作摘要
• 长期：运营人员的常用策略偏好和历史决策模式

💡 落地方案

  • "操作回忆"侧边栏：Pearl 内嵌 AI 侧边栏，自动汇总过去 1 小时的操作轨迹，切换回来时一句话告诉你"你刚才在看 XX 商家的 GMV 下滑数据"。
  • 策略推荐引擎：基于历史决策模式，相似场景出现时主动推荐策略。"上次遇到类似 GMV 下滑，你执行了 XX 促销策略，效果提升 15%"。
  • 跨系统剪贴板：数据看板复制的数据、飞书讨论的结论，自动同步到 Pearl 的 AI 助手。

💰 ROI 不高怎么办？—— 持续感知的成本优化

如果 always-on 感知的推理成本 ROI 不高，有哪些更聪明的做法？
👆 点击展开
Tiko 面对数十万商家同时在线的高并发场景，全量感知操作记录的推理成本极高。
核心问题：如果"持续感知"的 ROI 不高——大部分时候感知到的上下文并没有被用到——怎么办？

🔧 方案 1：事件驱动替代轮询

  • 不做 24/7 持续录制，监听关键事件：设置变更、订单异常、商家发起咨询。推理成本降低 90%+。

🔧 方案 2：分层缓存 + 摘要压缩

  • 热层（5 分钟内）：完整操作序列。温层（5-60 分钟）：小模型摘要为 1-2 句话。冷层（1h+）：仅保留关键词索引。
  • 每次对话上下文增量控制在 500-2000 tokens，成本几乎可忽略。

🔧 方案 3：用户反馈驱动的自适应感知

  • 初期最小感知，Tiko 答不好时用户点"不满意"触发深度回溯。系统逐步学习哪些场景值得 always-on。
  • 让数据告诉你 ROI，避免拍脑袋。可复用 Tiko 现有的 ORR 评测 + Bad Case 闭环机制来验证效果。

🧠 更大的想象空间：还能借鉴什么？

跳出"操作记忆"，Agent 自主感知的范式还能怎么用在 Tiko 和 Pearl 上？
👆 点击展开

💡 想法 1：Tiko "预判式"客服

  • 不等商家来问，Tiko 主动发现问题。监控 Seller Center 操作模式，检测到异常（反复修改同一设置、某页面停留过长），主动弹出"需要帮助吗？"。
  • Chronicle 是"看到你在做什么，下次你问我已经懂了"；Tiko 可以是"看到你卡住了，我主动来帮你"。
  • 从被动客服变成主动运营助手，直接提升 NPS 和解决率。

💡 想法 2：Pearl "运营 Copilot" —— 从工具到搭档

  • 借鉴 Claude Code Routines "可复用工作流"：运营日常操作（巡检数据→发现异常→执行策略→记录结果）封装为运营 Routine。
  • Pearl AI Copilot 观察操作模式，自动生成 Routine 模板："检测到你每周一都做 XX 巡检，要不要我帮你自动化？"
  • 与 Tiko Web 端正在开发的"定时任务"能力天然对接。

💡 想法 3：跨商家知识自动沉淀

  • Screenpipe 的"全量记录+检索"映射到跨商家知识库：A 商家遇到的物流问题，解决方案自动沉淀，B 商家遇到类似问题时 Tiko 直接引用。
  • 目前 Tiko 的 TTS 知识库是人工维护 + 知识点自动提取。如果能从真实对话中自动提取和沉淀解决方案，可大幅提升知识库覆盖率和更新频率。
  • 与 Tiko 现有的知识库质量检测（冲突识别、有效性检查）配合使用。

💡 想法 4：合规友好的"操作影子"

  • 电商场景涉及交易数据和商家隐私，不能全量录制。做"操作影子"：只记录操作类型和对象 ID，不记录具体数值。
  • 例如记录"商家 A 修改了物流模板 B 的配送区域字段"，不记录具体改成什么值。需要时通过 ID 实时查询。
  • 既获得操作上下文关联能力，又避免存储敏感交易数据。

📌 行动建议

短期（1-2 月）：Tiko 接入 Seller Center 操作日志，新增"操作上下文 Skill"，实现会话触发+按需回溯。预计减少追问轮次 30%+，提升 ORR。

中期（1 季度）：Pearl 上线"操作回忆"侧边栏+策略推荐引擎。Tiko 用 ORR 评测+Bad Case 闭环验证哪些场景值得 always-on 感知。

长期（2-3 季度）：构建"运营 Routine"自动化框架+跨商家知识自动沉淀，把 Tiko 和 Pearl 从工具升级为 AI 搭档。
