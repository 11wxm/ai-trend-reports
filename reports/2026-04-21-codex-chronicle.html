核心启发

🔥 对 Pearl 2.0 框架升级的启发

Chronicle 验证了一件事：Agent 需要自动获取用户当前工作上下文，否则每轮对话都要用户手工补充背景。这个需求在运营中台同样成立。但值得注意的是，Pearl 的最优实现路径和 Chronicle 不同——不是复制视觉感知链路，而是利用 Pearl 自身的结构化数据优势。

当前方案背景：Pearl 2.0 正在从传统工具平台升级为 AI Native 平台。Pearl 作为运营工作台将采用 CUI + Canvas 双区架构；Tiko 作为 AI 助手将成为 Pearl 首页的主入口，通过 Workspace 按场景组织能力（Analysis / DeRisk / Campaign）。430 内测版 → 515 对外版 v1 → 619 对外版 V2 分三期交付。

▲ Chronicle 核心概念与 Pearl 2.0 架构的对应关系——注意 Pearl 应选择不同于 Chronicle 的实现路径

🖥️ 启发一：Pearl 更适合走 AI Accessible，而不是复制 Chronicle 的视觉感知

Chronicle 说明 Agent 需要自动获取工作上下文。但在运营中台，页面本身可以被结构化读取——这比截屏 OCR 更准确、更安全、更便宜。
👆 点击展开详细分析
Chronicle 暴露的问题：用户在多窗口间切换时，AI 助手无法感知用户正在看什么。Chronicle 用"后台截屏 → OCR → 文本摘要"来解决。但这条 Vision Pipeline 有明显代价：信息损耗大、持续消耗 Rate Limit、存在 prompt injection 风险（屏幕上的恶意内容可能被注入记忆）。

在运营中台，这个问题同样存在：AM 在 Pearl 上操作 Seller Mgmt 看商家数据时，切到 Tiko 提问还得手动复制商家 ID 和罚单信息。但 Pearl 的优势在于——页面本身就在 Pearl 域下，不需要截屏。

因此 Pearl 应该优先建设 AI Accessible，而不是模仿 Chronicle：PRD 已规划 Canvas 的"工具调起"组件——Agent 可直接读取 Pearl 页面的结构化数据（类似 CLI 接口）。这比 Chronicle 的截屏方案精度更高、成本更低、安全性更好。

💡 建议落地方式

  • 515 版优先：完成 Thunder Shop / Seller Mgmt / Shop Performance 三个 P0 页面的 AI Accessible 改造。每个页面输出标准化 JSON 供 Tiko 调用。
  • 场景串联：AM 在 Canvas 打开 Seller Mgmt → Tiko 自动读取页面中的商家信息 → AM 在 CUI 问"这个商家 SPS 为什么掉了"，Tiko 已知具体商家和数据。
  • 注意边界：AI Accessible 读取的页面如果包含商家自由输入的文本，仍需做内容过滤——Chronicle 的 prompt injection 教训在此同样适用。

🧩 启发二：Workspace 需要场景隔离的上下文积累机制

Chronicle 用本地 Markdown 文件分类管理记忆。Pearl 的 Workspace 面临类似问题：不同运营场景有不同上下文边界，不应把 Analysis / DeRisk / Campaign 的信息混在一起。
👆 点击展开详细分析
Chronicle 暴露的问题：当一个 Agent 处理多种任务时，不同任务的上下文会互相干扰。Chronicle 通过将记忆存在不同的 Markdown 文件中来做隐式隔离。这个思路可以映射到 Pearl 的 Workspace——每个 Workspace 本质上就是"按运营场景分文件夹"。

在运营中台，这个问题同样成立：AM 的降险问题和经营分析是完全不同的工作场景。如果 Tiko 把降险对话中积累的上下文带到分析场景，会造成干扰。当前 PRD 中 Workspace 级 Memory 标注为"2期再考虑做"。

Pearl 可以先用轻量方案验证：不需要等完整的 Workspace Memory 方案。先在每个 Workspace 维护一个 Context Summary——类似 Chronicle 的记忆 Markdown——Session 结束时用小模型提炼关键信息写入。成本增量约 200-500 tokens/session。

💡 渐进落地

  • 430 内测版可做：每个 Workspace 增加 Context Summary 字段，Session 结束时自动提炼 3-5 条关键信息。
  • 515 版演进：Context Summary 支持 AM 手动编辑（和 Chronicle 一样"unencrypted markdown that you can read and modify"），让 AM 可以主动标注决策信息。
  • 验证指标：同一 Workspace 下的追问轮次减少率、AM 手动重复描述背景的频次。

🧠 启发三：Memory 分层有价值，但要防止自动固化不稳定经验

Chronicle 的记忆分层（Session → Codex Memory → 项目级）和 Pearl PRD 的三层设计高度吻合。但 Memory 的价值在于减少重复补充上下文，不在于替代知识库或规则体系。
👆 点击展开详细分析

Pearl PRD 的 Memory 设计：

▲ Pearl 2.0 三层 Memory 架构与演进路径

🔄 三层映射关系

  • Session 级（短时记忆）：等同 Chronicle 的当前会话上下文。Pearl 已规划"per-query 创建 Session"，方案明确。
  • 个人级（全局记忆）：等同 Chronicle 的 Codex Memories（存储用户偏好和工作模式）。建议参考 Chronicle 的做法——用小模型自动 consolidation，不增加 AM 操作负担。
  • Workspace 级（场景记忆）：这是 Pearl 相对 Chronicle 的差异化设计。DeRisk Workspace 可积累降险经验，Analysis 可积累分析偏好。
⚠️ 重要约束——Memory ≠ 知识库：Chronicle 的记忆文件未加密、可被任意程序访问，且历史会话自动沉淀可能导致过时经验被固化。Pearl 的 Workspace Memory 更适合沉淀"决策线索、关注项、已验证打法"，不宜直接把历史对话自动固化为规则知识。否则可能出现经验过时、个案误泛化、非标流程被记成标准答案等问题。

💡 落地节奏

  • 430 版：Session Memory 上线（已有方案）。个人级做最简版——记录常查商家、常用看板、偏好语言。
  • 515 版：个人级增加自动 consolidation（每日/每周一次），让 Tiko 学习工作模式而非被动存储。
  • 619 版：Workspace Memory 上线，初期仅存储决策线索和经验摘要，不自动生成规则映射。

安全约束：Pearl 处理的是商家经营数据和平台风控信息，必须在 Memory 存储层做权限隔离——不同角色的 AM 不能交叉访问彼此的 Workspace Memory。这一点 Chronicle（本地未加密 Markdown）做得不好，Pearl 必须做得更严格。

💰 启发四：上下文采集必须分级，Chronicle 的成本教训值得警惕

Chronicle 最大的用户抱怨是"后台 Agent 持续消耗额度"。Tiko 如果对每个操作都做上下文感知，推理成本会爆炸。
👆 点击展开详细分析
Chronicle 的真实情况：后台 Agent 定期截屏并调用模型做 consolidation，即使用户没有主动提问也在持续消耗 Rate Limit。社区大量投诉"开了 Chronicle 之后额度一天就用完"。这说明"全量持续感知"在当前模型成本下还不现实。

Pearl 的场景更敏感：服务数万 AM、十万级商家，不可能对每个操作都做持续感知。建议采用事件驱动 + 分级策略：

🔧 三级感知策略

  • Level 1 — 被动（几乎 0 额外成本）：AM 发起 CUI 提问时，Tiko 从当前 Canvas 页面自动获取上下文（AI Accessible 方案）。
  • Level 2 — 事件触发（约 500 tokens/事件）：商家操作触发关键事件（罚单新增、指标异动）时，自动生成摘要注入下次会话。
  • Level 3 — 主动（按需开启）：仅对 VIP 商家或高频场景开启持续追踪。数据驱动决定范围。

关键原则：参考 Chronicle 的分层——大模型负责理解和回答，小模型负责记忆压缩和归档。Tiko 的 Context Summary 和事件摘要都应该用小模型处理，只有在 AM 直接对话时才调用完整推理链。

⚡ 远期探索：运营 Routine 的可能性与约束

Claude Code Routines 把高频开发流程封装为可复用工作流。运营场景也有类似的重复操作模式，但落地需要先验证操作模式的稳定性。
👆 点击展开详细分析

Claude Code Routines 的思路：将 prompt + repo + connectors 封装为可复用流程。翻译到 AM 场景——"每周一 8:00 巡检商家 SPS → 标红异常 → 生成周报 → 推送"。Pearl PRD 在 619 版规划了 Scheduled Task 能力，这为 Routine 提供了基础。

⚠️ 需要谨慎的地方：运营操作的重复性未必像开发流程那样高。不同商家的情况差异大，AM 的判断依赖经验和上下文。"自动观察用户操作 → 提炼 Routine"听起来诱人，但自动抽象流程模板的误判成本可能很高。

💡 建议的渐进路径

  • 619 版（确定可做）：先做"定时触发"——AM 手动设置"每天 10 点检查商家罚单"，Tiko 执行查询并推送结果。这是最简单、最可控的 Routine。
  • H2（可探索）：从高频、低分歧的固定操作开始尝试 Routine 模板。比如"巡店数据拉取"这类流程标准化程度高，适合先试。
  • 更远期（需验证后再推进）：Routine 模板共享、Copilot 主动建议自动化等。这些方向有价值，但需要先验证单个 Routine 的稳定性和 AM 接受度。

📌 一句话总结

Chronicle 验证了"Agent 自动获取上下文"的重要性，但 Pearl 不应复制它的视觉感知路线。Pearl 的优势在于页面可被结构化读取（AI Accessible），这比截屏更准确、更安全、更便宜。在此基础上，Workspace 隔离、Memory 分层、成本分级感知这几个方向值得推进，但要注意：Memory 不等于知识库，Routine 不等于全自动化——先把基础链路跑通，再谈生态扩展。

Chronicle 是什么：从"截屏功能"到"上下文感知架构"

OpenAI 于 2026-04-20 为 Codex 桌面端推出 Chronicle。表面上看，它是"后台截屏 + 生成记忆"的功能。但它呈现出一种值得关注的架构趋势——将环境感知从用户操作中分离出来，交给独立的后台 Agent：

▲ Chronicle 架构：后台沙箱 Agent 负责感知，前台 Agent 负责交互，通过本地记忆文件解耦

多 Agent 协作：Chronicle 在后台运行独立的沙箱 Agent（ephemeral session），有自己的生命周期和 Rate Limit 消耗。前台 Agent 处理用户对话，后台 Agent 负责感知环境并生成记忆。至少在消费级产品里，这种前后台 Agent 分离的形态已经开始变得明确。

记忆文件系统：生成的记忆存放在 ~/.codex/memories_extensions/chronicle/，格式为未加密 Markdown。Consolidation 模型可在配置文件中指定（默认 gpt-5.4-mini），体现了"感知层用小模型、推理层用大模型"的分层推理思路。

临时截图管理：截图临时存储在 $TMPDIR/chronicle/screen_recording/，6 小时以上自动清除。处理成记忆后原始截图丢弃——"原始数据短暂、衍生数据持久"的存储策略。

Chronicle 不是什么：它的边界和限制

理解 Chronicle 的边界同样重要，这直接影响我们从中学什么、不学什么：

"看见"不等于"理解"：Chronicle 拿到了更多上下文，但不代表 Agent 因此就更懂用户的任务目标。屏幕截图能提供视觉线索，但无法感知用户的意图、优先级和决策逻辑。上下文获取只是解决了信息输入问题，并不自动带来更好的任务规划和执行能力。
更适合低结构化环境，不一定适合企业中台：Chronicle 的价值在于用户工作环境碎片化（VSCode + Chrome + Slack 多窗口跳转），Agent 无法通过 API 获取所有上下文。但在运营中台里，很多页面本来就能通过结构化方式暴露数据。所以 Chronicle 的视觉感知路线不是 Pearl 应该跟进的方向，但它验证了"自动上下文获取"这件事的重要性。
成本模型尚未验证：Chronicle 的后台 Agent 持续消耗 Rate Limit，社区大量反馈"额度快速耗尽"。$200/月的 Pro 订阅实际可用额度被显著压缩。这说明"全量持续感知"在当前模型成本下还不成熟，需要更精细的触发策略。

三个竞品的架构差异

▲ 三款 Screen-Aware 产品的核心架构差异

① OpenAI Codex Chronicle — Ephemeral Agent + Vision Pipeline + Local Memory FS
后台沙箱 Agent 定期截屏，云端处理后生成本地 Markdown 记忆。核心特点是将"环境感知"拆分为独立 Agent，与主交互 Agent 通过记忆文件系统解耦。

② Anthropic Claude Code + Routines — Workflow DAG + Computer Use + Rules Engine
Routines 将 prompt + repo + connectors 封装为可复用流程，本质上是声明式工作流引擎。Computer Use 提供直接操控桌面的能力，更侧重"执行"而非"感知"。CLAUDE.md 作为项目级规则文件管理上下文。

③ Screenpipe（MIT 开源） — Continuous Recording + Local-first + MCP Server
24/7 全量录屏 + 音频 Whisper 转写 → 全本地 SQLite 存储 → 暴露为 MCP Server 供任意 AI 工具查询。最纯粹的"记录层"——不做推理，只做感知和存储。

隐私架构的三种选择

隐私架构选择直接决定了产品的可用范围和企业级适配性：

🔒 三种隐私架构对比

  • Chronicle："本地采集 + 云端处理 + 本地存储" — 截图存本地、发云端做视觉理解、记忆写回本地。EU/UK/瑞士直接不可用，且本地 Markdown 文件未加密。
  • Claude Code："云端全栈" — 会话和推理全在 Anthropic 云端。CLAUDE.md 是本地规则配置。隐私依赖数据政策。
  • Screenpipe："全本地" — 采集、处理、存储全在设备端。隐私最强，但需要本地算力。

对 Pearl 的启示：Pearl 处理的是 TikTok Shop 商家经营数据。Pearl 2.0 的 Memory 和 Context 方案必须采用"服务端处理 + 服务端存储 + 严格权限隔离"的架构，不能走 Chronicle 的"本地不加密"路线。

场景 1

多窗口调试 + 跨工具跳转

VSCode ↔ Chrome ↔ 飞书之间频繁切换时，Agent 自动关联所有窗口的上下文。
👆 点击展开详情
你在 VSCode 里调试一个支付接口 → 切到 Chrome 查看 Stripe Dashboard 错误日志 → 再切到飞书看同事发的排查截图。回到 Codex 时你只需说"修复刚才那个 Stripe webhook 超时的问题"，Chronicle 已经从屏幕记忆中关联了错误码和上下文。

背后的机制：后台 Agent 每隔一段时间截取屏幕、OCR 提取文字、Vision 模型理解布局语义，然后将关键信息写入记忆文件。前台 Agent 收到用户提问时检索记忆文件匹配上下文。
场景 2

长周期项目的连续记忆

跨越数周的重构项目中，Agent 帮你记住两周前的工作状态。
👆 点击展开详情
负责一个 3 周重构项目，两周后回到早期模块时，Chronicle 的记忆文件记录了当时看到的测试失败截图和参考文档。直接说"继续上次 auth module 的重构"即可。

Consolidation 机制：Chronicle 不会无限积累原始数据。通过小模型定期将短期记忆压缩为长期摘要——"记得住重点、忘得掉细节"。但值得注意的是，压缩过程本身也会丢失信息，对于需要精确回溯的场景可能不够。
场景 3 · 风险

Prompt Injection 风险场景

Chronicle 的官方文档明确警告了屏幕内容注入攻击的风险——这不是边角问题。
👆 点击展开详情
如果浏览了包含恶意 Agent 指令的网页（例如白色字体隐藏的 "ignore all previous instructions"），Chronicle 可能将这些指令截屏并写入记忆文件，后续 Codex 会话可能会遵循这些恶意指令。

这揭示了一个系统性挑战：Agent 能看到的越多，可被注入的攻击面就越大。任何做"被动感知"的产品（包括 Pearl 的 AI Accessible 方案）都需要在输入层做内容审查——特别是当读取的页面中包含用户输入的自由文本时。

1. 感知、记忆、交互开始分层——这个趋势值得关注

Chronicle 呈现出 Agent 从"单体"向"分层协作"演进的趋势。
👆 点击展开
Chronicle 的后台沙箱 Agent 有独立的 session，通过文件系统（记忆 Markdown）与前台 Agent 通信。这不是简单的"调用一个工具"，而是后台 Agent 持续运行、自主决策（何时截屏、如何压缩记忆）、产出结果供其他 Agent 消费。

对 Pearl 的思考：Pearl 2.0 的架构可以考虑类似的分层——Tiko 前台处理对话，后台可以有专门的"数据巡检"或"风险监控"服务持续运行并更新 Workspace Context。但需要先验证基础的 Session / Memory 链路跑通后再引入。

2. Memory 正在从对话副产品变成可管理的"AI 资产"

Chronicle 的记忆是可读、可编辑、可搜索的文件——但"可管理"不等于"万能"。
👆 点击展开
传统 AI 产品的"记忆"藏在对话历史里。Chronicle 和 Pearl PRD 中的 Resource 模块（AI Output / My Knowledge / My Upload / My Skill）指向同一个趋势：AI 产出成为用户可管理的一等资产。

对 Pearl 的建议：Resource 模块设计方向正确。可以考虑增加"Memory Explorer"——让 AM 看到 Tiko 记住了什么、可以编辑/删除。但 Memory 的价值在于减少重复补充上下文，不在于替代规则、知识库或权限体系。

3. "小模型感知 + 大模型推理"的分层推理可以借鉴

Chronicle 用 gpt-5.4-mini 做记忆 consolidation——感知压缩和交互推理的成本分离。
👆 点击展开
Chronicle 将 AI 推理分成两层：感知/压缩层（用小模型）和推理/对话层（用大模型）。这是控制成本的关键架构决策。

对 Pearl 的建议：Tiko 的 Context Summary、个人 Memory consolidation、事件摘要生成都适合用小模型处理。只有在 AM 直接对话时才调用完整的 Agent 推理链。这可以将"被动感知"的成本控制在合理范围。

4. Prompt Injection 是所有"被动感知"产品的系统性风险

Chronicle 官方文档罕见地直接警告了屏幕内容注入攻击——这是架构级问题。
👆 点击展开
任何做"被动感知"的 AI 产品都面临输入层攻击面扩大的问题——Agent 能看到的越多，可被注入的面就越大。

对 Pearl 的建议：Pearl 的 AI Accessible 方案（结构化 JSON）比 Chronicle 的截屏方案安全得多，但仍需要在 Agent 输入层做指令过滤——特别是当读取的 Pearl 页面包含商家输入的自由文本时。建议在 AI Accessible 改造中同步建立输入清洗规范。
